# API 變更說明 — 2026-07-20

臨打團球類/分級改版、使用者 `username`、以及訂單刪除。本篇整理**前端需要調整**的重點，含破壞性變更 (Breaking) 標記。

> 對應 OpenAPI：`docs/openapi.yml`（Swagger UI 已同步）。

| # | 變更 | Breaking |

## 1. 公開臨打團清單 `GET /pickup-groups`、`GET /hosts/{host_id}/pickup-groups`

**選擇性登入**：端點維持公開，但現在若帶上有效的 `Authorization: Bearer <token>`，回應會針對該使用者填入 `enrolled_status`；未登入則一律 `free`。建議前端在使用者已登入時一律帶上 token 以取得正確狀態。

**回應項目 (`PickupGroupBrief`) 變更 — Breaking：**

| 舊 | 新 |
|----|----|
| `host_nickname` | 移除 → 改用 `host_id` + `host_username` + `host_display_name` |
| `skill_level`（字串 `"B"`）| `skill_level` 物件 `{ id, name }` |
| —（無）| 新增 `sport` 物件 `{ id, code, name }` |
| —（無）| 新增 `start_time`、`fee` |
| —（無）| 新增 `enrolled_status`：`free` / `pending` / `confirmed` / `cancelled` / `cancel_request` |

**過濾參數變更 — Breaking：**
- ~~`?skill_level=A`~~ → `?sport_id=<uuid>`、`?skill_level_id=<uuid>`

範例回應項目：

```json
{
  "id": "…",
  "host_id": "…",
  "host_username": "coach_wang",
  "host_display_name": "王教練",
  "location_id": "…",
  "title": "週日早晨羽球",
  "sport": { "id": "…", "code": "BADMINTON", "name": "Badminton" },
  "skill_level": { "id": "…", "name": "B" },
  "start_time": "2026-07-21T01:00:00Z",
  "fee": 200,
  "enrolled_status": "pending"
}
```

## 2. 球類 (Sports) 與 分級 (Skill Levels)

球類與分級改為**由系統管理員維護的資料表**，並開放公開查詢。分級**隸屬單一球類**，未來可依球類設定不同分級。

### 新端點

| 方法 | 路徑 | 權限 |
|------|------|------|
| GET | `/sports`、`/sports/{id}` | 公開 |
| POST / PATCH / DELETE | `/sports`、`/sports/{id}` | 系統管理員 |
| GET | `/skill-levels`（支援 `?sport_id=`）、`/skill-levels/{id}` | 公開 |
| POST / PATCH / DELETE | `/skill-levels`、`/skill-levels/{id}` | 系統管理員 |

- `Sport`：`{ id, code, name, is_active, … }`。`code` 為穩定機器碼（自動轉大寫，如 `BADMINTON`）。
- `SkillLevel`：`{ id, sport_id, name, sort_order, is_active, … }`。
- 刪除為**軟刪除**（`is_active=false`）。
- 建立臨打團的下拉選單流程：先 `GET /sports` → 依選定球類 `GET /skill-levels?sport_id=<sportId>`。
- 系統已預先種入 `BADMINTON` / `BASKETBALL` / `VOLLEYBALL`，各含 `A`/`B`/`C`/`D` 分級。

### 建立 / 更新臨打團 — Breaking

`POST /pickup-groups`、`PATCH /pickup-groups/{id}` 的 body 變更：

| 舊欄位 | 新欄位 |
|--------|--------|
| `skill_level`（`"A"`）| 移除 |
| `host_name`、`host_phone` | 移除（主辦資訊改由 `users` 帶入）|
| —（無）| `sport_id`（必填, uuid）|
| —（無）| `skill_level_id`（必填, uuid）|

- 驗證：`skill_level_id` 必須隸屬於 `sport_id`，否則回 `400`；球類或分級不存在/停用亦回 `400`/`404`。

### `GET /pickup-groups/{id}`（`PickupGroupResponse`）— Breaking

- 移除頂層 `host_id`、`host_name`、`host_phone`、`skill_level`（字串）。
- 新增 `host` 物件：`{ id, username, display_name, phone }`（`phone` 即時取自 `users`）。
- 新增 `sport`、`skill_level` 巢狀物件（同上）。

```json
{
  "id": "…",
  "host": { "id": "…", "username": "coach_wang", "display_name": "王教練", "phone": "0912…" },
  "title": "小團",
  "start_time": "…", "end_time": "…",
  "fee": 100, "capacity": 8,
  "location_id": "…",
  "sport": { "id": "…", "code": "BADMINTON", "name": "Badminton" },
  "skill_level": { "id": "…", "name": "C" },
  "status": "active", "enable": true,
  "current_enrolled": 3,
  "created_at": "…", "updated_at": "…"
}
```

## 3. 刪除報名訂單 `DELETE /pickup-orders/{id}`（新增）

- 權限：**該臨打團主辦人或系統管理員**（報名者本人不可，仍走 `PATCH` 改 `cancelled`/`cancel_request`）。
- 成功回 `204`；非主辦/管理員回 `403`；訂單不存在回 `404`。
- 硬刪除；刪除非 `cancelled` 狀態的訂單後，該團 `current_enrolled` 會自動 -1。

## 4. 使用者 `username`

- `UserResponse` 新增 `username`（唯一帳號代稱）。
- **註冊 `POST /auth/register` 新增必填欄位 `username`** — Breaking：
  - 格式：`^[a-z0-9_]{4,15}$`（小寫英文、數字、底線，4–15 字）。
  - 大寫會**自動轉小寫**（`CoolUser_1` → `cooluser_1`）。
  - 格式錯誤回 `400`；重複（比對時不分大小寫）回 `409`。
- 目前 `username` **不可修改**（未放入 `PATCH /users/{id}`）。
- 既有使用者已由後端遷移自動配發隨機 `username`，前端顯示時直接使用即可。

註冊 body 範例：

```json
{
  "email": "user@example.com",
  "username": "cool_user",
  "password": "password123",
  "display_name": "小明"
}
```

## 5. 訂單 `rejected` 狀態與「拒絕」流程

新增報名生命週期狀態 `rejected`，讓主辦人以「拒絕」取代「硬刪除」來移除報名者。

**訂單狀態新增值 — Breaking：**

- `PickupOrderResponse.status`、`UpdatePickupOrderRequest.status`、`PickupGroupBrief.enrolled_status` 皆新增 `rejected`。
- 前端若有狀態列舉 (enum) 或狀態文案對照表，需補上 `rejected`（建議顯示為「已拒絕」）。

**主辦人拒絕報名者：`PATCH /pickup-orders/{id}` body `{ "status": "rejected" }`**

- 權限：僅 group host 或系統管理員可設定 `rejected`；報名者本人不可（本人只能 `cancelled` / `cancel_request`，嘗試設 `rejected` 回 `403`）。
- 效果：訂單保留（不刪除）、`rejected` 不計入 `current_enrolled`（等同釋放名額），且該使用者**無法再次報名同一團**（再次 `POST /pickup-groups/{id}/orders` 回 `409`）。

**硬刪除權限縮緊 — Breaking：`DELETE /pickup-orders/{id}` 改為僅限系統管理員**

| 角色 | 舊 | 新 |
|------|----|----|
| 主辦人 (host) | 可硬刪除 → `204` | **不可**，改用拒絕 → `403` |
| 系統管理員 | 可硬刪除 → `204` | 維持可硬刪除 → `204` |

- 系統管理員硬刪除後，該訂單完全移除，該使用者**可**再次報名（與拒絕不同）。

**名額釋放時機調整 — Breaking：**

- `cancel_request`（申請取消中）**仍占用名額**，須待訂單真正變成 `cancelled` 後才釋放。
- 只有 `cancelled` 與 `rejected` 會被排除在 `current_enrolled` 之外。
- （先前 `cancel_request` 會立即釋放名額，此版本起改為保留至真正取消。）

**重新報名規則：**

| 現有訂單狀態 | 再次報名同一團 |
|------|------|
| `pending` / `confirmed` / `cancel_request` | `409`（仍占用名額，視為已報名） |
| `cancelled` | ✅ 允許（沿用同一筆訂單，`id` 不變，狀態重置為 `pending`） |
| `rejected` | `409`（已被拒絕，永久擋下） |
